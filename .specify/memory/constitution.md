# Save Token Kit 项目章程

## 核心原则

### I. CLI 优先

每个功能模块通过 CLI 命令暴露。命令行工具 `stk`(save-token-kit) 是用户与系统交互的唯一入口。

- 所有核心能力必须可通过 `stk <command>` 调用。
- CLI 参数解析遵循 POSIX 规范：`stk <verb> [options] [args]`。
- 输出遵循文本协议：结果输出到 stdout，错误输出到 stderr。
- 支持 `--json` 标志输出机器可读格式，默认输出人类可读格式。
- `--help` 和 `--version` 必须对每个命令可用。

**理由**: CLI 是 AI Agent 可编程调用最可靠的方式。文本 I/O 确保可调试性、可组合性和跨平台兼容性。

### II. Token 效率优先

项目核心使命是为 AI Agent 节省 Token 消耗。每个设计决策必须权衡 Token 成本。

- 配置文件、Skill 文件、Prompt 文件的内容必须精简，去除冗余描述。
- 输出格式默认紧凑；详细模式仅通过 `--verbose` 标志启用。
- 命令输出应避免装饰性内容(如彩色框、重复标题)，除非显式要求。
- 新增功能必须评估其对上下文窗口的影响——避免引入不必要的长文本输出。
- 项目自身的 Agent 指令(Commands/Skills)也应遵循 Token 节省原则，作为产品的最佳实践示范。

**理由**: 项目存在的意义就是降低 Token 消耗。自身的膨胀会削弱产品的说服力。

### III. 测试驱动质量

单元测试覆盖度 MUST >= 60%，集成测试覆盖关键路径。

- 新增功能 MUST 包含对应的单元测试。
- 关键 CLI 命令 MUST 有集成测试覆盖(端到端调用验证)。
- 测试框架：Vitest。
- 测试文件命名：`*.test.ts`，与源文件同目录或置于 `tests/` 对应路径。
- CI 流水线 MUST 在合并前运行全部测试，失败则阻塞合并。

**理由**: CLI 工具的可靠性直接影响 AI Agent 的工作流稳定性。充分的测试是用户信任的基础。

### IV. 简洁至上

代码力求简洁，遵循 YAGNI(You Aren't Gonna Need It)原则。

- 拒绝为"将来可能"的需求预留扩展点。
- 单个函数不超过 50 行(测试函数除外)。
- 不引入超过当前需求范围的抽象层(如不必要的接口、工厂模式)。
- 优先使用标准库和轻量依赖，避免重量级框架。
- 复杂逻辑 MUST 有英文注释解释"为什么"，而非"是什么"。

**理由**: 简洁的代码更易维护、更少 bug、Token 占用更低。项目作为工具链的一环，不应成为复杂度的来源。

### V. 文档即产品

面向用户的文档使用中文，代码注释使用英文。

- README.md、用户指南、命令帮助文本使用简体中文。
- 代码注释(含 JSDoc)使用英文。
- 每个 CLI 命令 MUST 有 `--help` 输出，包含中文描述和示例。
- 架构决策 MUST 记录在 `.specify/` 相关文档中。
- CHANGELOG.md 使用中文记录每个版本的变更。

**理由**: 目标用户主要是中文开发者，中文文档降低使用门槛。英文注释保持代码的国际化可读性。

## 技术栈约束

- **语言**: TypeScript，严格模式(`strict: true`)。
- **运行时**: Node.js >= 18 LTS。
- **包管理**: pnpm。
- **构建工具**: tsc 或 tsx。
- **测试框架**: Vitest。
- **代码规范**: ESLint + Prettier(业界通用配置)。
- **版本管理**: Changesets(参考 OpenSpec-cn 实践)。
- **CI/CD**: GitHub Actions。

## 开发工作流

- 功能开发从 `spec` 文档开始，遵循 OpenSpec 规范驱动流程。
- 分支命名: `###-feature-name`，与 spec 编号对应。
- 提交信息遵循 Conventional Commits 格式(中文描述)。
- PR MUST 通过 CI 检查(测试 + 代码规范)后方可合并。
- 代码审查关注：测试覆盖、Token 效率、简洁性。

## 治理

本章程是项目开发的最高准则，所有设计决策和实施活动 MUST 遵循本章程。

- 章程修订 MUST 通过 PR 进行，并记录修订理由。
- 任何偏离章程原则的设计决策 MUST 在 PR 中明确说明理由并获得批准。
- 版本号遵循语义版本控制(SemVer):
  - MAJOR: 原则删除或不兼容的重新定义。
  - MINOR: 新增原则或章节。
  - PATCH: 措辞澄清、拼写修正。
- 合规性在每个 spec 的计划阶段通过"章程检查"门控验证。
