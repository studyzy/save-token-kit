# 功能规范: 上下文感知的 /stk:analyze 优化分析

**功能分支**: `002-context-aware-analyze`
**创建时间**: 2026-07-13
**状态**: 草稿
**输入**: 用户描述: "我希望 /stk:analyze 能实现根据我项目的实际情况，如我的使用目的：代码编写，文档写作，通用办公？当前如果是代码仓库，代码和相关文档是否都在这个仓库，从而提出优化建议。我之前写了一版，你可以参考 ../save-token 的 analyze"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 分析时收集项目上下文 (优先级: P1)

用户运行 `/stk:analyze` 或 `stk analyze` 后，工具先了解用户**使用目的**和**项目布局**，再产出优化建议。当前版本只基于诊断数据（工具/Skill/MCP 占用）按节省量排序给出建议，忽略用户实际在做什么，导致给文档写作用户也推一堆代码审查类精简建议，建议不贴切。

**优先级原因**: 这是"上下文感知"的核心——没有上下文收集，后续所有场景化判断都无从谈起。没有它就没有价值增量。

**独立测试**: 在任意代码仓库运行分析，工具会通过交互（或本地缓存）先询问并记住使用目的与代码/文档是否同仓，随后落盘的分析结果里能体现这些上下文字段，且每条建议带 `scenario` 归因。

**验收场景**:

1. **给定** 一个空的 `/stk:analyze` 调用，且本地无缓存上下文, **当** 分析启动, **那么** 工具会向用户收集"使用目的"与"代码和文档是否在同一仓库"两项信息后再产出建议。
2. **给定** 用户此前已回答过上下文问题并被缓存, **当** 再次运行分析, **那么** 工具复用缓存、不再重复询问（除非用户显式指定 `--reask` 或提供新上下文）。
3. **给定** 用户回答使用目的="文档写作", **当** 分析完成, **那么** 产出的建议集中排除明显仅适用于代码开发的条目（如代码审查 Agent 精简），并优先推荐文档协作类优化。

---

### 用户故事 2 - 按使用目的裁剪优化建议 (优先级: P2)

分析引擎根据用户声明的使用目的（代码编写 / 文档写作 / 通用办公），对诊断出的原始优化空间做过滤、重排和措辞调整，使建议与用户实际工作匹配。

**优先级原因**: 直接决定建议的"贴切度"，是用户感知到价值的关键差异点。

**独立测试**: 同一份诊断数据，分别声明三种使用目的运行分析，三次产出的建议集合明显不同（条目取舍与排序不同），且每类场景的高亮建议符合其用途。

**验收场景**:

1. **给定** 诊断发现仓库含大量未引用 MCP, **当** 使用目的="通用办公", **那么** 建议重点为"禁用无关 MCP / 精简 Skill"，而非代码特定优化。
2. **给定** 诊断发现 presentation 类 Skill 占用 300 token, **当** 使用目的="代码编写", **那么** 该条目被标记为可删并给出高优先级。
3. **给定** 使用目的="文档写作", **当** 分析产出, **那么** 报告中明确标注 `scenario: docs` 并优先推荐文档类（如知识库、写作辅助 MCP）优化。

---

### 用户故事 3 - 识别代码/文档同仓并给出专项建议 (优先级: P3)

当项目既是代码仓库、且代码与文档同在一处时，针对"文档也可能被当作上下文反复读取"的情况给出专项节省建议（如将文档移出主上下文、建立文档索引/知识库、排除文档目录的自动读取等）。若代码与文档分离（文档在独立仓库），则建议把文档从本仓库上下文排除。

**优先级原因**: 这是用户明确点名的具体洞察点，能产出其他工具给不了的专属建议，但依赖故事 1 的上下文数据。

**独立测试**: 在一个代码与文档同仓的仓库运行分析并声明"同仓"，报告应出现至少一条针对"文档与代码同仓导致的重复上下文读取"的专项建议；声明"分离"时出现"排除文档仓库上下文"类建议。

**验收场景**:

1. **给定** 用户声明"代码和文档在同一仓库", **当** 分析完成, **那么** 报告包含针对同仓文档冗余读取的专项建议（如建立文档索引、限制文档自动注入）。
2. **给定** 用户声明"文档在独立仓库", **当** 分析完成, **那么** 报告建议将文档仓库排除出主代码对话上下文。
3. **给定** 项目非代码仓库（纯文档/办公）, **当** 分析完成, **那么** 不产出代码特定建议，仅围绕文档与办公场景。

---

### 边界情况

- 用户拒绝回答上下文问题 → 回退到现有"通用分析模式"，仍产出基于诊断数据的建议，不阻塞。
- 诊断数据缺失（未运行 diagnose）→ 提示先运行 `/stk-diagnose`，与现有行为一致。
- 用户中途切换使用目的（如项目从编码转向文档）→ 提供重问机制刷新上下文缓存后再分析。
- 同一仓库多人/多用途共用 → 上下文以"最近一次回答"为准，允许用户随时重问覆盖。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须在分析开始时收集用户上下文，至少包含"使用目的"（代码编写 / 文档写作 / 通用办公）与"代码与文档是否在同一仓库"两项。
- **FR-002**: 系统必须缓存已收集的上下文，并在后续分析中默认复用，避免每次重复询问。
- **FR-003**: 系统必须允许用户覆盖/重问上下文（如 `--reask` 参数或显式声明），刷新缓存后重新分析。
- **FR-004**: 系统必须依据使用目的对诊断出的优化空间进行过滤与重排，使建议与用途匹配。
- **FR-005**: 系统必须为每条产出的建议标注其适用场景归因（如 `scenario` 字段），便于用户理解为何提出。
- **FR-006**: 系统必须在代码与文档同仓时，产出针对文档冗余上下文读取的专项优化建议。
- **FR-007**: 系统必须在代码与文档分离时，产出将文档仓库排除出主上下文的建议。
- **FR-008**: 系统必须在上下文缺失或用户拒绝提供时，回退到通用分析模式且不阻塞分析流程。
- **FR-009**: 系统必须将最终建议落盘至 `./save-token/analysis.json`，并在结构中保留上下文字段（使用目的、同仓标记）供下游 `/stk-optimize` 复用。
- **FR-010**: 系统必须保持"只产出建议、不修改任何文件"的边界，与现有 analyze 行为一致。

### 关键实体 *(如果功能涉及数据则包含)*

- **ProjectContext（项目上下文）**: 用户声明的使用目的（code/docs/office）、代码与文档是否同仓（同仓/分离/不适用）。可缓存于本地。
- **ContextualSuggestion（上下文感知建议）**: 在现有优化建议基础上附加 `scenario` 归因与"为何针对此场景"的说明。
- **AnalysisReport（分析报告）**: 继承现有 `analysis.json` 结构，新增 `context` 字段承载 ProjectContext，供下游复用。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 90% 的运行在首次收集上下文后，后续运行不再重复询问即可直接产出建议。
- **SC-002**: 对同一份诊断数据，分别声明三种使用目的时，产出的建议条目集合差异率 ≥ 30%（条目取舍或排序不同）。
- **SC-003**: 当用户声明"代码与文档同仓"时，报告中 100% 出现至少一条专项同仓建议；声明"分离"时 100% 出现排除文档上下文建议。
- **SC-004**: 拒绝提供上下文时，100% 回退到通用模式并成功产出建议，不报错阻塞。
- **SC-005**: 落盘的 `analysis.json` 在结构校验中 100% 包含 `context` 字段，且不破坏现有下游 `/stk-optimize` 消费。
