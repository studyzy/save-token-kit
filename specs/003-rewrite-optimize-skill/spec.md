# 功能规范: 重写 stk-optimize SKILL 支持分级优化

**功能分支**: `003-rewrite-optimize-skill`
**创建时间**: 2026-07-14
**状态**: 草稿
**输入**: 用户描述: "我希望重写 src/templates/skills/stk-optimize/SKILL.md 能够先读取save-token/tasks.md 然后询问用户的Token优化等级：初级，初级+中级，初级+中级+高级全部。然后根据用户的选择，筛序出待办任务列表，然后按顺序逐个执行优化任务。如果是第三方工具的安装和启用，可以调用stk命令 stk install XXX -g --agent codebuddy 来安装。"

## 用户场景与测试 _(必填)_

### 用户故事 1 [分级筛选后执行优化] (优先级: P1)

用户希望在应用优化建议前，先了解并控制优化的"激进程度"。他运行 `/stk-optimize`，Agent 读取 `save-token/tasks.md`，询问他希望执行哪个等级（初级 / 初级+中级 / 全部），然后只执行被选中等级的待办任务，且按列表顺序逐项落地。

**优先级原因**: 这是本功能的核心价值——把"一次性全部应用高风险修改"变为"用户可逐步、可控地优化"，降低误改风险，也是与原 SKILL 最大的行为差异。

**独立测试**: 准备一份含初级/中级/高级任务的 `tasks.md`，触发 `/stk-optimize`，选择"初级"，验证只执行初级任务；选择"全部"，验证三级任务都执行。

**验收场景**:

1. **给定** `save-token/tasks.md` 存在且含三级任务，**当** 触发 `/stk-optimize` 并选择"初级"，**则** 仅初级任务被筛选进入执行队列，中级/高级被跳过。
2. **给定** 同上，**当** 选择"初级+中级"，**则** 初级与中级任务进入队列，高级被跳过。
3. **给定** 同上，**当** 选择"全部"，**则** 初/中/高三级任务全部进入队列。
4. **给定** 队列已确定，**当** Agent 开始执行，**则** 任务严格按 `tasks.md` 中出现的顺序逐个执行。
5. **给定** 某任务为第三方工具安装/启用，**当** 进入执行，**则** 通过 `stk install <工具名> -g --agent codebuddy` 完成安装与启用。

### 用户故事 2 [无 tasks.md 的兜底] (优先级: P2)

当用户尚未生成 `tasks.md` 时，流程不应静默失败，而应给出明确指引。

**优先级原因**: 保证流程健壮性，避免用户面对空输出不知所措。

**独立测试**: 在不存在 `tasks.md` 的目录下触发 `/stk-optimize`，验证 Agent 提示用户先运行诊断/分析生成 `tasks.md`，而非报错退出。

**验收场景**:

1. **给定** `save-token/tasks.md` 不存在，**当** 触发 `/stk-optimize`，**则** Agent 提示用户先执行 `/stk-analyze` 等前置步骤以生成 `tasks.md`，并停止。

### 用户故事 3 [执行结果落盘] (优先级: P3)

执行过程中每个任务的成功/失败状态需被记录，便于后续复查与报告。

**优先级原因**: 与原 SKILL 行为保持一致，保证优化可追踪、可复盘。

**独立测试**: 执行若干任务后，检查 `save-token/tasks.md` 或配套文件中的状态标记是否反映实际执行结果。

**验收场景**:

1. **给定** 任务队列开始执行，**当** 某任务完成或失败，**则** 其状态（completed / failed）被写入产出物，失败任务附带错误信息。
2. **给定** 某任务执行失败，**当** Agent 继续，**则** 记录失败并继续下一任务（或按需停下，见边界），不阻塞整体流程可见性。

## 功能需求

### 需求 1：读取并解析分级待办清单

系统必须读取项目根目录下 `./save-token/tasks.md`，解析其中的优化任务及其所属等级（初级/中级/高级）。

- FR-1.1: 若 `tasks.md` 不存在，提示用户先生成（见用户故事 2）。
- FR-1.2: 每个任务必须带有行内等级标识 `[初级]` / `[中级]` / `[高级]`，位于 `- [ ] ` 复选框之后（格式：`- [ ] [中级] 描述`）。
- FR-1.3: 任务保留其在文件中的顺序，作为默认执行顺序。

### 需求 2：询问并接收优化等级

系统必须向用户呈现三个等级选项，并接收其选择：

- 初级
- 初级 + 中级
- 全部（初级 + 中级 + 高级）

- FR-2.1: 选项须覆盖上述三种组合，无遗漏。
- FR-2.2: 选择后，系统基于等级对任务做筛选。

### 需求 3：按等级筛选并执行

系统必须依据用户选择，仅保留对应等级的任务，并按原顺序逐个执行。

- FR-3.1: 选择"初级" → 仅执行初级任务。
- FR-3.2: 选择"初级+中级" → 执行初级与中级任务。
- FR-3.3: 选择"全部" → 执行初/中/高全部任务。
- FR-3.4: 执行顺序严格遵循 `tasks.md` 中的出现顺序。
- FR-3.5: 描述以"保持/保留/当前配置"开头且预估节省为 0 的任务视为 no-op，标记为 `skipped`，不执行任何修改。

### 需求 4：第三方工具安装/启用的专用路径

当被筛选的任务属于"第三方工具安装/启用"类型时，系统必须通过命令 `stk install <工具名> -g --agent codebuddy` 完成安装与启用。该命令为真实 CLI 子命令（非提示词）。

- FR-4.1: 工具名从注册表（`src/tools/`）解析，匹配任务描述中的工具标识（如"Headroom"→`headroom`，"RTK"→`rtk`）。
- FR-4.2: `stk install <tool> [-g|--global] [--agent <name>]` 行为：
  - `-g/--global`：安装到全局 `~/.codebuddy/`（默认全局）；`--local` 覆盖为项目级 `.codebuddy/`。
  - `--agent <name>`：默认 `codebuddy`；`claude`/`codex` 等预留，未支持时显式报错"暂不支持"。
  - 执行顺序：注册表查工具 → 校验 agent 支持 → 运行 `installCommand` → 运行 `configCommand`（注入 agent/global）→ 输出结果。
- FR-4.3: 该路径区别于本地配置修改类任务（如禁用 skill、裁剪 CODEBUDDY.md），后者仍按原 SKILL 的备份+修改方式执行。

### 需求 5：执行状态记录

系统必须将每个任务的执行结果写入产出物。

- FR-5.1: 成功标记 completed，失败标记 failed 并附错误。
- FR-5.2: 产出物位置遵循项目约定（`./save-token/` 目录）。

## 成功标准

- SC-1: 在含三级任务的 `tasks.md` 下，选择某一等级后，实际执行的任务集合 100% 等于该等级对应的任务集合（无一遗漏、无越级）。
- SC-2: 任务执行顺序与 `tasks.md` 中定义顺序一致（偏差为 0）。
- SC-3: 第三方工具类任务 100% 通过 `stk install <工具名> -g --agent codebuddy` 触发，无手动步骤遗漏。
- SC-4: 当用户未生成 `tasks.md` 时，100% 给出明确前置指引而非静默失败。
- SC-5: 每个被执行任务的最终状态（成功/失败）100% 被记录到产出物。

## 关键实体

- **tasks.md**：人类可读的优化待办清单，含任务描述、等级、目标与状态。位于 `./save-token/tasks.md`。
- **优化等级**：初级 / 中级 / 高级，用于筛选任务集合。
- **优化任务**：一条可执行的 Token 节省操作（禁用 skill、裁剪配置、安装第三方工具等）。
- **第三方工具**：可通过 `stk install` 安装启用的外部省 Token 工具（如 rtk、caveman、headroom 等）。

## 假设

- A-1: `tasks.md` 由前置的 `/stk-analyze` 流程生成，或由用户手工维护，SKILL 自身不负责生成该文件。
- A-2: 任务的等级以文件内行内标签 `[初级]`/`[中级]`/`[高级]` 为准，位于复选框 `- [ ] ` 之后。
- A-3: 第三方工具的安装命令统一为 `stk install <工具名> -g --agent codebuddy`，工具名可从任务目标字段解析。
- A-4: 本地配置修改类任务沿用原 SKILL 的"先备份再修改"原则。

## 边界

- 操作失败处理：单个任务失败应记录并继续，整体流程可见；如需遇错即停，可在执行前与用户确认（沿用原 SKILL 边界精神）。
- 本 SKILL 仅指导 Agent 执行修改，不自动反向回滚；回滚依赖用户侧备份手动恢复。
- 仅支持 `codebuddy` 平台。

## 等级标注方式（已确认）

`tasks.md` 由 `/stk-analyze` 前置生成，每个任务以列表项呈现，等级用行内标签标注：

```markdown
- [ ] [中级] 禁用 skill: ponytail-help（预估节省 ~50 Token）
      原因：帮助类 Skill，代码场景非高频
- [ ] [高级] 启用 Headroom（预估节省 ~6200 Token）
      原因：已安装未启用
```

- 标签位置：复选框 `- [ ] ` 之后，格式为 `[初级]` / `[中级]` / `[高级]`。
- 文件尾部含 `等级统计：初级 N 项 / 中级 M 项 / 高级 K 项` 供快速核对。
- "保持现状/无节省"类任务（如"保持 X 当前配置"，预估节省 0）属 no-op，应识别为可跳过项，标记为 `skipped` 而非执行修改。
